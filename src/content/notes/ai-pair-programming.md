把 AI 当成聪明、博学但会犯错的高级实习生：产出快，但不了解你的代码库，也不为后果负责。下面几条把它的产出变成可信代码。

## 给足上下文与约束

模型只看得见你给的东西。它不知道你的目录结构、技术选型和团队规范，缺失的部分会被"合理地编造"——这正是幻觉来源。动手前先喂料：

- 相关代码片段、类型定义、接口签名，而非口头描述
- 明确约束：语言/依赖版本、是否允许引新依赖、错误处理与命名规范
- 期望的输入输出示例，把边界情况也列出来

约束越具体，它越没有空间瞎猜。

## 任务切小，先方案后代码

别丢一句"帮我实现登录模块"。大任务里每一步小错会累积，最后你得读几百行去 debug，常比自己写还慢。把任务拆到单个函数或单次改动的粒度，并且**先让它讲方案，再写代码**：

```
先别写代码。说明你打算怎么实现这个限流器：
用什么数据结构、并发怎么处理、有哪些边界情况。
给 2 个方案并说明取舍，等我确认再写。
```

方案阶段纠错几乎零成本，代码返工就贵了；让它解释思路，也能暴露它是否真懂需求。

## Review、测试、不盲信

> 你按下回车那一刻，这段代码就是你写的，出 bug 也是你的责任。

生成的代码要当 PR 来审，按它的强弱分配注意力：

| 可放手 | 必须复核 |
|---|---|
| 样板 / CRUD / 格式转换 | 业务逻辑是否真对 |
| 写单元测试 | 边界、并发、错误路径 |
| 解释报错、读陌生代码 | 是否调用了不存在的 API |
| 正则、SQL 草稿 | 安全与性能含义 |

几个高杠杆用法：让它**为自己的代码写测试**，但你要审测试是否在真正断言；把报错堆栈整段贴给它做初步定位；让它逐行解释你看不懂的旧代码。核心心态——它出初稿，你判对错；不验证就合入，等于让实习生直接推生产。
