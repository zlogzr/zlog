模型读不到你的代码库，也读不懂你脑中的隐含约束。它只能在你塞进上下文窗口的那些 token 上推理。产出的上限由输入质量决定，而不是由模型有多聪明决定。

## 喂什么，而不是喂多少

上下文窗口是有限预算，每个 token 都在挤占其他 token 的注意力。原则是**高相关、高密度、低噪音**，按这个顺序取舍：

- **任务与约束**：要做什么、不能动什么、技术栈版本、必须遵守的接口签名。
- **关键代码**：直接相关的函数、类型定义、调用方。贴定义而非整个文件。
- **示例**：一个「输入 → 期望输出」的样例，比三段文字描述更能锁定格式。
- **反例/边界**：「不要用全局变量」「这个分支已废弃」。

> 不相关的信息不是中性的，它是负的。

## 噪音如何拖垮产出

无关代码、过时注释、互相矛盾的需求会稀释信号。模型不知道哪些该忽略，会一视同仁地推理：于是抄了你贴进去的废弃写法，或纠结一个与任务无关的细节。

| 做法 | 效果 |
|---|---|
| 贴整个文件求「改一下」 | 抓不住重点，改动发散 |
| 只贴目标函数 + 它依赖的类型 | 聚焦，改动可控 |
| 三个需求堆在一段话里 | 顾此失彼 |
| 拆成有序清单 | 逐条可核对 |

## 大知识库：先检索再生成

知识库塞不进窗口时，就先检索后生成：把文档切块、建索引，每次只取与当前问题最相关的几块拼进提示。关键不是「能塞多少」，而是「检索得准不准」——召回了噪音，等于主动往上下文里投毒。所以检索质量要单独评估，别只盯着生成结果。

一个可直接套用的拼装模板：

```text
约束：沿用现有 error 包装风格，勿引入新依赖。
任务：给 ParseConfig 增加对 timeout 字段的校验（<=0 报错）。
相关代码：
  func ParseConfig(...) (*Config, error) { ... }   // 当前实现
  type Config struct { Timeout int `json:"timeout"` }
期望：返回形如 fmt.Errorf("invalid timeout: %d", v) 的错误。
```

动手前先问一句：「这个任务最少需要哪几样信息？」这一步省下的，是后面反复纠偏的时间。
